Self-paced (no interval, e.g. /loop <prompt>): Claude does a chunk, picks its own gap (1m–1h) based on whether work remains, and stops itself when the goal is met. Use for every “Stop when X” loop. Fixed interval (/loop 1d …, 30m): polls on a clock and never self-stops. A ⟳ Routines chip means the loop wants unattended/overnight running — back it with a cloud Routine, since /loop needs an open session and expires after 7 days.

Clock times aren’t a /loop option — ask in natural language instead: “every day at 3am, …” and Claude schedules it via cron.

▸ /loop vs /goal — keep working until done

Both keep the session going across turns without re-prompting, and both can stop on their own. The difference is what starts the next turn and who decides you're done. Two questions decide it:

1. Is the work waiting on something external?a build/CI, a deploy, logs to appear, or a time of day
Yes →
/looppoll on an interval; nightly/unattended → a cloud Routine
No ↓
2. Can you state a clear pass/fail end condition?tests green · coverage 100% · queue empty · file under N lines
Yes →
/goalruns back-to-back; a second model verifies the condition
No ↓
/loop (self-paced)let it iterate and stop itself — open-ended work with no crisp finish line

One line: self-paced /loop waits between rounds and trusts itself; /goal runs flat-out with a second model as judge. So /goal is tighter for a local grind to a checkable end state (“coverage is 100%”), while /loop wins when there's genuinely something external to wait for. Write the goal so the proof lands in the chat (“npm test exits 0”) — the evaluator only reads the transcript, it can't run commands. Requires v2.1.139+.

Should you combine them? Usually no — most /goal + /loop “combos” are redundant and resolve to a single mechanism. The one real nest is /loop <interval> /goal <condition> (“periodically drive to a checkable end state”); the reverse doesn't compose. Quick compiler:

You say…It's…
“until <a condition I can check>”/goal
“every <time>” / “wait for <external thing>”/loop
“every <time>, drive <condition> to done”/loop … /goal … (rare)
both words, one taskyou meant one
